Writing code is only half the battle. Working effectively with a team requires maintaining a clean history, descriptive branch management, and collaborative review structures.

Section 01

Descriptive Branching Strategies

In a team environment, multiple developers work on the same codebase simultaneously. We use branching to isolate work and prevent breaking the production-ready code.

GitHub Flow Lifecycle

The standard collaboration cycle relies on short-lived feature branches merged into the main production branch via Pull Requests.

main
v1.0.0 (a1b2c3d)
Merge Pull Request (f8e9d0c)
feature/auth
feat: add login page (c4d5e6f)
feat: support Google Sign-in (b7a8c9d)
test: add auth flow tests (e5f6g7h)

Hover over nodes in the graph to inspect commit hashes and messages.

✓ Good Branch Practices
  • Keep branch names prefix-scoped: feat/user-auth, bugfix/refresh-token, docs/readme-updates.
  • Create separate branches for every separate task.
  • Delete branches on the remote repository after they are merged.
  • Pull updates from main daily to handle conflicts incrementally.
✗ Bad Branch Practices
  • Committing directly to the main or master branch.
  • Creating ambiguous, generic branch names: updates, fix, my-branch, test1.
  • Keeping merged branches on remote forever.
  • Combining unrelated features in one single branch.
Section 02

Commit Hygiene & Message Guidelines

A Git commit is more than a file backup; it is a ledger entry. Clean, structured commits make tracking bugs and understanding history simple.

Conventional Commit Structure

A highly accepted standard in modern software teams is the Conventional Commit specification: type(scope): brief message.

Types of Commits:

  • feat: Adding a new feature
  • fix: Patching a bug
  • docs: Writing documentation
  • style: Whitespace, formatting
  • refactor: Code rewrite (no features/fixes)
  • test: Adding or fixing tests
✓ Good Commits
  • git commit -m "feat(ui): add dashboard dark mode"
  • git commit -m "fix(api): resolve CORS preflight errors"
  • Keep commits atomic (one logical feature per commit).
  • Verify code runs before committing.
✗ Bad Commits
  • git commit -m "fix stuff"
  • git commit -m "temp work in progress"
  • git commit -m "completed page layout and hooked API database and added authentication"
  • Committing broken code that prevents build compiler execution.
Section 03

Pull Request Etiquette

Pull Requests (PRs) represent where code review, validation, and collaboration occur. A good PR is easy to review, quick to verify, and well-described.

A Perfect Pull Request Description Template

Provide context so reviewers don't have to read every line of code to understand your intent.

Markdown Description Template 
## Description
Brief summary of the changes introduced by this pull request.

## Related Issues
Closes #123

## How Has This Been Tested?
- [ ] Tested locally on Chrome & Firefox.
- [ ] Added integration tests for authentication.

## Screenshots (if applicable)
[Paste screenshots here]
✓ Good PR Etiquette
  • Keep PRs small (under 200 lines of code changes is ideal).
  • Run tests and lint check tools locally before opening a PR.
  • Write comments on your own code diff to explain tricky logic to reviewers.
  • Accept review feedback constructively.
✗ Bad PR Etiquette
  • Opening massive PRs with 1000+ line changes.
  • Leaving the description text blank or writing "Fixed issues."
  • Merging a branch before tests/CI builds successfully run.
  • Taking constructive code reviews personally.
