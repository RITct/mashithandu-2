A Brief History

Understanding how Git and GitHub came to shape modern software engineering.

2005

Linus Torvalds invents Git

Frustrated by licensing changes in BitKeeper (the tool used to manage the Linux kernel), Linus Torvalds designed Git in under two weeks. He aimed for speed, decentralization, and robust branch management.

2008

GitHub is Launched

Tom Preston-Werner, Chris Wanstrath, P. J. Hyett, and Scott Chacon built GitHub using Ruby on Rails. They wanted to make hosting and sharing repositories effortless, turning Git into a social coding network.

2018

Microsoft Acquires GitHub

Microsoft acquired GitHub for $7.5 billion. Since the acquisition, GitHub has made private repositories free for all users and expanded into CI/CD, cloud environments, and AI collaboration.

2020+

GitHub Actions & Copilot

With the integration of GitHub Actions (native automation/CI-CD) and GitHub Copilot (AI-assisted coding), GitHub transformed from a repository hosting platform into a comprehensive developer lifecycle hub.

Git vs. GitHub

They are often confused, but they serve completely different roles in version control and collaboration.

Git (The Local Engine)

A distributed version control system running on your local machine. It tracks history, handles branches, and lets you undo changes without needing any internet connection.

GitHub (The Cloud Hub)

A cloud-based hosting service for Git repositories. It adds a web interface, collaborative code reviews (Pull Requests), project tracking, issue discussions, and CI/CD pipelines.

Initialize a Repository

Prepares your current folder to track changes and creates a hidden .git folder where the local database resides.

git init

Run this once at the start of a new project. After running, you can create branches, check status, and save local commits.

Clone an Existing Repository

Downloads a copy of an entire project repository from a remote server (like GitHub) to your local computer.

git clone https://github.com/username/repository.git

This configures the origin remote link automatically. It creates a folder matching the repo name, downloads all historical commits, and checks out the default branch.

Check Status of Changes

Displays the status of the files in your directory: which files have changes that are staged, unstaged, or untracked.

git status

This is the safest command to run frequently. It helps you see what state your workspace is in and what you should do next (e.g., stage or commit changes).

Stage Your Changes

Moves changes from your Working Directory to the Staging Area, registering them for the next commit snapshot.

git add .

Using . stages all files in the current folder. You can also specify single files (e.g. git add index.html) to choose exactly what goes into the commit.

Record a Commit Snapshot

Saves your staged snapshot into your local repository's history timeline. Includes a short message explaining the change.

git commit -m "feat: implement hero section layout"

Always use descriptive commit messages. Every commit receives a unique SHA-1 identifier key, allowing you to rollback or reference it later.

Push Commits to Remote

Uploads all local branch commits to the remote repository (e.g. GitHub), sharing your changes with the team.

git push origin main

Here, origin represents the remote server URL, and main is the destination branch. This syncs your local history timeline with the cloud.

Pull Latest Remote Changes

Downloads commits from the remote server and merges them directly into your current local active branch.

git pull origin main

Run this before starting any work to make sure you have the latest updates from your team. This helps prevent merge conflicts later.
